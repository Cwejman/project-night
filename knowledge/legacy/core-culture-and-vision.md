## Layer 1

The project's culture is the approach itself: being unbound by context, replacing the cycle of context-dependence and second-hand memory with cyclically integrated, routine-based retrieval — RAG not as a bolt-on but as continuous living routine woven into every agent cycle. The system is simultaneously the POC that proves this approach, the container for knowledge about what's being built, and the tool that helps manifest it into creation. It bootstraps itself into existence — every entry about how it should work IS the system working. This is why lossless matters beyond engineering: every insight lost is a piece of the vision that must be re-discovered. The bigger picture is an atomic, vector-based system where every knowledge mutation is versioned like git — but for semantic content. Current files and directories are the grounding; the POC proves concepts the final system inherits.

## Layer 2

### Why this project exists

Not to build a documentation tool, not to add memory to Claude, not to implement RAG. To prove — and in proving, create — a fundamentally different relationship between agents and knowledge. One where:

- The session is a compute surface, not a memory surface
- Knowledge is cyclically integrated, not retrieved on demand
- Memory is first-class, not second-hand
- Context limits are a non-problem because knowledge lives outside the window
- Every insight persists — nothing is lost to session boundaries or compaction

### The self-building property

The system contains knowledge about what it should become. It uses that knowledge to guide its own development. Each session adds to both the system's capability and its self-understanding. The entries about architecture ARE the architecture. The entries about rules ARE the rules (and increasingly, the code that enforces them).

This is not circular — it's bootstrapping. The system pulls itself into existence through its own operation.

### The naming gap

The approach needs a name. Current descriptions orbit around:
- "Unbound by context"
- "RAG as routine" (not retrieve-then-generate, but continuous cyclical integration)
- "Memory-first" (not memory as afterthought)
- "Session-irrelevant" (the session is disposable, knowledge is permanent)

No name has been chosen. The culture is clear; the label is not.

### Core and integration — initially one

Platform integration (Claude Code hooks, TUI, bootstrap routines) is deeply tied to core at this stage. Separating them early creates artificial distance without benefit. They start as one body of knowledge with the acknowledgment that as the system grows and supports multiple platforms, separation may become valuable. Embedding is about excluding what's unnecessary — if integration knowledge is always needed alongside core, they belong together until proven otherwise.

### Scratch space IS a project

No separate "sandbox" concept needed. Open a new directory, launch Claude, the system integrates first-class — that's already both a scratch space and a project. It doesn't write back to core unless you choose to. The default project posture (bare files, no write-back to core) IS the scratch space. No need to separate these two ideas.

### The bigger picture — direction, not destination

There are clues throughout the system pointing toward something: atomic versioned mutations, semantic diffability, vector-based retrieval. The direction has truth in it but the exact form is not settled. The current filesystem + markdown approach is a deliberate grounding — simple enough to reason about, structured enough to prove concepts. What comes next will be refined through continued exploration. Those connected to the vision will recognize which clues are aligned. Big picture is never lost — that is part of the lossless nature and the core values.
