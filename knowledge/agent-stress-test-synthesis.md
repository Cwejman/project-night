# Agent Stress Test Synthesis — 8 Agents, 4 Domains, A/B Comparison

> Generated by Claude from 8 parallel agent results. To be reviewed and iterated with the author.

## The Experiment

8 agents modeled the dimensional knowledge system against 4 domains. Each domain had two agents:
- **A (minimal):** Use only existing primitives (chunks, dimensions, weights, meta-chunks, scope). No extensions.
- **B (extensions allowed):** Same domain, may propose minimal extensions if primitives feel insufficient.

Domains tested: Culture (night project), Organization (OpenLight non-profit), Software (the CLI tool), Website (OpenLight public site).

## Universal Findings (All 8 Agents Agree)

### 1. The core model works for knowledge navigation

Every agent found that scope-as-query, dimension-to-dimension navigation, and weight-based relevance work naturally. Cross-cutting queries are the model's strength. The model is genuinely better than hierarchical or relational approaches for exploring interconnected knowledge.

### 2. Instance-of vs relates-to is the primary gap

**All 8 agents** — both A and B approaches, across all 4 domains — identified this as the most significant tension. The model has one relationship type (weight on dimension). But two fundamentally different things happen:
- "Alice IS a person" (instance/membership)
- "This methodology chunk RELATES TO people" (topical/associative)

Without distinguishing these, every collection query ("list all projects," "show all team members," "what bugs exist?") is contaminated by topical content.

**The A agents** (no extensions) all named this as the deepest strain and could not work around it satisfactorily. Weight thresholds are fragile. Double-scoping doesn't solve it. The model genuinely lacks this primitive.

**The B agents** all proposed the same fix: a minimal quality marker on weights. The exact vocabulary varied:
- Culture-B: `instance-of`, `describes`, `exemplifies`, `contrasts-with`
- Org-B: `instance-of`, `about`, `aspect-of`
- Software-B: `instance`, `describes`, `affects`
- Website-B: `instance`, `relates`

**Convergence:** The minimum viable set is **two kinds**: `instance` and `relates` (or `about`). Software-B's `affects` (causal) is interesting but may be domain-specific. Culture-B's four kinds may be overcomplicated. The binary distinction resolves the core tension.

### 3. Structured metadata is needed for operational domains

**Culture** (pure knowledge) found metadata marginal — almost everything is conceptual.

**Organization, Software, and Website** all found it essential:
- People need names, emails, phone numbers, roles
- Projects need budgets, dates, statuses
- Dependencies need version numbers, licenses
- Code needs file paths, function names

Without structured fields, rendering a team directory requires NLP parsing of prose. What a traditional DB does trivially (SELECT name, email FROM people) requires an agent in this model.

**The fix:** An optional structured payload/metadata map alongside chunk text. Not dimensionally weighted — just queryable key-value pairs. The text carries meaning; the payload carries data.

**Note:** This is needed when the system serves operational interfaces (websites, directories). For pure knowledge browsing, it may not be needed.

### 4. Dimension properties are NOT needed

**All B agents** explicitly considered and rejected dimension-level properties (like "collection" vs "topic"). The instance/relates quality on weights makes this unnecessary — a dimension with many `instance` chunks IS a collection, emergently. No declaration needed.

## Domain-Specific Findings

### Culture Domain

**Works naturally for:** Metaphor-to-principle connections, cultural compression (the three-beat sentence), cognitive slices, self-building/autopoietic content.

**Strains at:**
- **Prescriptive vs descriptive** — rules ("write-first") and metaphors ("night") are structurally identical but functionally different. Rules override; metaphors inform.
- **Conflict resolution** — no "supersedes" mechanism when chunks contradict.
- **Temporal evolution** — the three-beat sentence evolved from earlier formulations. The model sees them as concurrent chunks, not as evolution.
- **Dimensional granularity** — in a holistic domain where everything connects, there's no principled stopping point for how many dimensions to create.
- **Weight assignment is subjective** — especially for metaphorical content. Different humans would assign different weights.

### Organization Domain

**Works naturally for:** Cross-cutting queries ("what connects community and energy?"), values as cross-cutting dimensions, project navigation, emergent connections between topics.

**Strains at:**
- **Instance-of** (severe) — team page, projects page, partners page all break without it.
- **Structured records** (severe) — names, emails, phone numbers, dates don't fit as prose.
- **Sibling disambiguation** — the model can't distinguish "sunbridge and clearwater are siblings (both projects)" from "sunbridge and energy are related topics."
- **Scope-as-AND can't do union** — "all people = staff OR board OR advisors" needs three separate queries or umbrella dimensions.
- **Referential integrity** — when someone leaves a project, multiple weight adjustments across multiple chunks, no transaction guarantees.

### Software Domain

**Works naturally for:** Cross-cutting concern navigation, design decision traceability, bug triage by concern area, exploratory browsing.

**Strains at:**
- **Containment hierarchy** (module > function > parameter) — intersection ≠ descent. The model can't express "everything inside the storage module."
- **Causal chains** — "this design decision caused this bug" requires directed relationships, not co-occurrence.
- **Entity state lifecycle** — bug open → closed. Weight shift from "bugs" misrepresents — the chunk is still ABOUT bugs, it's just resolved.
- **Code references** — file paths, line numbers, function names need exact-match retrieval, not semantic similarity.
- **Scope negation** — "everything about storage that is NOT a bug" needs subtraction, which the model doesn't have.
- **Dimension promotion threshold** — when should a chunk become a dimension? No principled guidance.

### Website Domain

**Works naturally for:** Pages-as-scopes (genuinely elegant), cross-cutting queries, content evolution with safety, the "Ask OpenLight" visitor query feature, blog posts as dimension clusters.

**Strains at:**
- **Page ordering/navigation** — views are unordered. Navigation menus need explicit ordering.
- **SEO metadata** — title tags, og:images, structured data are per-view machine data, not knowledge.
- **Visual layout** — "this chunk should be the hero block" is a per-view rendering decision, not a dimensional property.
- **Content stability** — published pages need snapshot stability, not live resolution. Solved by static generation pipeline.
- **Performance** — live knowledge resolution is too slow for public page loads. Pre-rendering needed.

**Website-B's key finding:** The website should be a **pre-rendered browser with cached views**. Static generation from knowledge views (like Next.js/Hugo but with the knowledge system as source instead of markdown files). Most pages static, "Ask OpenLight" live. The model supports this naturally through the view abstraction.

## The Extensions Worth Keeping

| Extension | Needed By | Impact on Model | Verdict |
|---|---|---|---|
| **Instance/relates quality on weights** | All domains with entities | Adds one marker to the weight primitive | **Essential. The minimum viable extension.** |
| **Structured metadata on chunks** | Org, Software, Website | Optional payload alongside text | **Essential for operational use. Can be deferred for pure knowledge.** |
| **Display settings on views** | Website | Zero model impact — pure interface layer | **Needed for rendering, not for the model itself.** |
| **Approval events + drift detection** | Website, any external dependency | New event type in atomic history | **Natural fit with existing model. Not a new primitive.** |
| **Scope negation** | Software | Adds subtraction to scope operations | **Worth considering. Minor addition to scope mechanics.** |

## What Remains Open

- **Weight assignment:** How are weights determined? All agents noted this is subjective. Auto-suggestion from embeddings? Calibration? Qualitative levels mapped to numbers?
- **Dimension lifecycle:** When does a chunk earn its own dimension? The threshold is felt but unprincipled.
- **Temporal ordering:** The model has atomic history (events in time) but no concept of content-level temporal ordering. Meeting minutes, blog posts, and events need dates that aren't dimensional weights.
- **Causality:** "This decision caused this bug" is a directed relationship. The model only has undirected co-occurrence through shared dimensions.
- **Prescriptive authority:** Some chunks are rules that override. The model has no authority hierarchy.
- **Scope union:** Scope-as-AND works for narrowing but not for "show me A OR B." Umbrella dimensions are a workaround but not a solution.

## What This Means for the PoC

The model's core — chunks, dimensions, weights, scope — is validated across all four domains. The scope-as-query mechanism is genuinely powerful.

Two extensions appear essential before building:
1. **Instance/relates** on weights (one binary marker)
2. **Structured metadata** on chunks (optional payload)

Everything else (display settings, approval events, scope negation) can be layered on later without changing the core model.
